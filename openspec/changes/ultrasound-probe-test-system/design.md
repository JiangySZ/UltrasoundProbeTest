## Context

本项目旨在开发一款Windows平台的超声探头测试系统，用于自动化测试、校准、ID烧录及报告生成。由于无实际硬件环境，需设计硬件模拟层以支持开发和测试。

### 现有约束
- **平台**: Windows 10/11 (64-bit)
- **Qt版本**: 5.9.1
- **编译器**: VS 2017 Community (MSVC)
- **PDF库**: Qtpdf
- **配置文件**: JSON格式
- **硬件**: 无实际硬件，需模拟

## Goals / Non-Goals

**Goals:**
- 实现完整的探头检测、激活、测试流程
- 支持表面回波测试和标准回波测试两种模式
- 实现自动化阵元扫描和DSP分析
- 生成符合FDA Traceability要求的PDF报告
- 支持探头ID烧录功能
- 构建可切换的硬件模拟层

**Non-Goals:**
- 不实现实际硬件驱动（使用模拟层）
- 不支持非Windows平台
- 不实现网络远程控制功能

## Decisions

### 1. 架构选择：分层架构
**决定**: 采用分层架构（UI层 / 业务层 / 硬件抽象层）  
**理由**: 清晰分离关注点，便于后续硬件驱动实现替换模拟层  
**备选**: MVC/MVP模式 - 对于Qt桌面应用过于复杂

### 2. UI框架：Qt Widgets
**决定**: 使用Qt Widgets而非Qt Quick  
**理由**: 更成熟的图表库支持（QCustomPlot），更适合测试仪器类应用  
**备选**: Qt Quick - 更现代，但图表生态较弱

### 3. DSP算法实现
**决定**: 使用FFTW库进行频域分析  
**理由**: 开源、高效、C++原生支持  
**备选**: Qt自带QSignalProcessing - 功能有限

### 4. PDF生成：Qtpdf
**决定**: 使用Qtpdf库  
**理由**: 与Qt原生集成，支持中文（需配置字体）  
**备选**: PDFTron - 商业授权成本高

### 5. 配置文件：JSON
**决定**: 使用JSON格式配置文件  
**理由**: 更易读易解析，Qt原生支持QJsonDocument  
**备选**: XML - 原需求定义，但JSON更现代化

### 6. 硬件模拟层设计
**决定**: 实现接口抽象层（HardwareInterface），提供Mock实现  
**理由**: 模拟层与真实驱动接口一致，便于切换  
**实现**: 
```
HardwareInterface (抽象基类)
├── RealHardwareDriver (真实驱动 - 未来实现)
└── MockHardwareDriver (模拟驱动 - 当前使用)
```

### 7. 自动化测试与自愈
**决定**: 集成自动化测试框架，实现持续集成与自修复  
**理由**: 减少人工介入，提升开发效率与代码质量  
**实现**:
- 使用Qt Test框架进行单元测试
- 集成CI/CD流水线（GitHub Actions或Jenkins）
- 实现自动化回归测试
- 配置自动代码检查（clang-tidy, cppcheck）
- 集成自动修复脚本（针对常见问题）

## Risks / Trade-offs

| 风险 | 描述 | 缓解措施 |
|------|------|----------|
| 硬件接口不确定性 | 无实际DLL，可能与规格不符 | 设计宽松的抽象层，允许参数调整 |
| PDF中文支持 | Qtpdf中文字体配置复杂 | 使用开源字体（如Noto Sans CJK） |
| DSP算法精度 | 模拟数据可能与实际差异大 | 提供可配置的容差参数 |
| 测试覆盖 | 无硬件难以完整测试 | 编写单元测试，验证算法逻辑 |
| CI/CD配置复杂 | Qt 5.9.1与新版本工具链兼容问题 | 使用兼容版本的CI工具 |
| 自动修复局限性 | 自动化修复无法覆盖所有场景 | 设置人工复核机制 |

## Open Questions

1. **DLL接口细节**: 实际硬件DLL的函数签名、参数类型尚未明确，需在开发过程中与硬件团队确认
2. **测试标准值**: 各类探头的灵敏度、带宽等标准值及容差范围需用户提供
3. **PDF模板**: 报告的具体格式和内容布局需确认
